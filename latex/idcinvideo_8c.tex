\doxysubsection{Detailed Description}
id Quake II Cin Video \mbox{\hyperlink{struct_decoder}{Decoder}} by Dr. Tim Ferguson For more information about the id C\+IN format, visit\+: \href{http://www.csse.monash.edu.au/~timf/}{\texttt{ http\+://www.\+csse.\+monash.\+edu.\+au/$\sim$timf/}}

This video decoder outputs P\+A\+L8 colorspace data. Interacting with this decoder is a little involved. During initialization, the demuxer must transmit the 65536-\/byte Huffman table(s) to the decoder via extradata. Then, whenever a palette change is encountered while demuxing the file, the demuxer must use the same extradata space to transmit an A\+V\+Palette\+Control structure.

id C\+IN video is purely Huffman-\/coded, intraframe-\/only codec. It achieves a little more compression by exploiting the fact that adjacent pixels tend to be similar.

Note that this decoder could use libavcodec\textquotesingle{}s optimized \mbox{\hyperlink{struct_v_l_c}{V\+LC}} facilities rather than naive, tree-\/based Huffman decoding. However, there are 256 Huffman tables. Plus, the \mbox{\hyperlink{struct_v_l_c}{V\+LC}} bit coding order is right -\/$>$ left instead or left -\/$>$ right, so all of the bits would have to be reversed. Further, the original Quake II implementation likely used a similar naive decoding algorithm and it worked fine on much lower spec machines.
