{\footnotesize\ttfamily I\+D3\+D11\+Texture2D$\ast$ A\+V\+D3\+D11\+V\+A\+Frames\+Context\+::texture}

The canonical texture used for pool allocation. If this is set to N\+U\+LL on init, the hwframes implementation will allocate and set an array texture if initial\+\_\+pool\+\_\+size $>$ 0.

The only situation when the A\+PI user should set this is\+:
\begin{DoxyItemize}
\item the user wants to do manual pool allocation (setting \mbox{\hyperlink{struct_a_v_h_w_frames_context_ad67ff3a5828a611d1b1cddd64fc1de71}{A\+V\+H\+W\+Frames\+Context.\+pool}}), instead of letting \mbox{\hyperlink{struct_a_v_h_w_frames_context}{A\+V\+H\+W\+Frames\+Context}} allocate the pool
\item of an array texture
\item and wants it to use it for decoding
\item this has to be done before calling \mbox{\hyperlink{hwcontext_8c_a66a7e1ebc7e459ce07d3de6639ac7e38}{av\+\_\+hwframe\+\_\+ctx\+\_\+init()}}
\end{DoxyItemize}

Deallocating the \mbox{\hyperlink{struct_a_v_h_w_frames_context}{A\+V\+H\+W\+Frames\+Context}} will always release this interface, and it does not matter whether it was user-\/allocated.

This is in particular used by the libavcodec D3\+D11\+VA hwaccel, which requires a single array texture. It will create I\+D3\+D11\+Video\+Decoder\+Output\+View objects for each array texture element on decoder initialization.
